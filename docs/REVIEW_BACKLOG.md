# 审查与重构清单

本文记录当前项目在代码结构、性能、可维护性和产品能力上的重点重构项。

## P0

### 1. LLM 协议适配层需要独立

当前 `LlmClient` 仍然把所有调用都压到 Chat Completions 兼容路径。  
建议抽象成：

- `openai_responses`
- `openai_chat_compatible`
- `openrouter`
- `bailian_responses_cn`

这样设置页、测试接口、分析任务和后续 agent flow 才不会继续耦合在一个实现上。

### 2. 分析任务需要升级成多阶段 agent workflow

当前 `AnalysisService` 是单任务、单 prompt、单结果。  
如果目标是深度研究式报告，建议拆成：

1. 市场上下文聚合
2. 个股数据清洗
3. 新闻/事件证据提取
4. 初稿生成
5. 反证与风险复核
6. 最终结构化报告

每个阶段都应可持久化。

## P1

### 3. 个股联动查询需要缓存或预索引

`AkshareProvider._get_concept_boards()` 现在会遍历概念板块并逐个拉成分股，属于典型 N+1 查询。  
单股偶尔可接受，但一旦批量分析或频繁打开详情页，会变慢。

建议：

- 加本地缓存
- 预构建“股票 -> 概念板块”索引
- 或单独做定时刷新任务

### 4. 分析任务与 UI 刷新策略还偏粗

个股页和自选页有多处固定轮询，容易导致：

- 无意义请求
- 背景窗口资源消耗偏高
- 任务与页面刷新的状态耦合

建议把：

- 行情刷新
- 分析队列刷新
- 提醒刷新

拆成更明确的前后台策略。

### 5. SQLite repository 仍偏“全量读 + 内存筛选”

目前一些查询还会先 `.all()` 再在内存中过滤。  
在数据量扩大后会变慢，也不利于索引设计。

建议后续：

- 把筛选条件尽量下推到 SQL
- 明确索引
- 为分析记录、提醒事件、自选条目分别做查询路径优化

## P2

### 6. 文档和能力边界需要持续同步

当前文档已经比之前完善，但后续每次改：

- 启动方式
- 协议支持
- 数据源能力
- 分析结构
- UI 工作台布局

都需要同步更新文档，否则很快又会和代码脱节。

### 7. 构建产物和工作区卫生要继续收敛

需要继续避免：

- 误提交 `*.tsbuildinfo`
- 重复目录
- 平台专属临时产物

建议后续补一套最基础的 CI 检查。

## 建议执行顺序

1. LLM provider adapter
2. 多阶段 agent workflow
3. 联动/板块缓存优化
4. 数据库查询优化
5. CI 与仓库卫生
